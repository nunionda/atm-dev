## 1. 프로젝트 개요

### 1.1 프로젝트 배경

개인 투자자의 주식 매매에서 반복적으로 발생하는 문제가 있다.
감정에 의한 비합리적 매매, 손절 타이밍 놓침, 시장 모니터링의 물리적 한계,
분산투자 원칙 미준수 등이 그것이다.

본 프로젝트는 이러한 문제를 **규칙 기반 자동매매 시스템**으로 해결하고자 한다.

### 1.2 목적

국내 주식(KOSPI/KOSDAQ) 현물 시장을 대상으로,
기술적 분석 기반의 매매 시그널을 자동 생성하고,
사전에 정의된 규칙에 따라 포트폴리오를 구성·관리·청산하는
자동매매 소프트웨어를 개발한다.

### 1.3 비전

| 현재 문제 (수동 매매) | 목표 상태 (자동매매 시스템) |
|------------------------|----------------------------|
| 감정에 의한 비합리적 매매 | 규칙 기반 일관된 매매 실행 |
| 시장 모니터링 시간 한계 | 장중 전 시간 자동 모니터링 |
| 손절 타이밍 놓침 (심리적 저항) | 기계적 손절/익절 실행 |
| 분산투자 원칙 미준수 | 체계적 포트폴리오 비중 관리 |
| 매매 기록 부재 | 전 매매 내역 자동 로깅 및 성과 분석 |

### 1.4 핵심 비즈니스 프로세스 (High-Level)

```
┌─────────┐    ┌─────────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 데이터  │    │ 시그널 생성 │    │ 의사결정 │    │ 주문실행 │    │ 사후관리 │
│ 수집    │ ─→ │ (진입/청산) │ ─→ │ & 리스크 │ ─→ │          │ ─→ │ & 로깅   │
│         │    │             │    │ 검증     │    │          │    │          │
└─────────┘    └─────────────┘    └──────────┘    └──────────┘    └──────────┘
 시세/호가/       기술적 분석        BUY/SELL/       증권사 API      매매 로그
 거래량 수집      전략 적용          HOLD 판단       주문 전송       성과 분석
                                   리스크 게이트                    알림 발송
```

> **아키텍트 노트**: 원래 요구사항에서 "수익률 분석 → 시그널 발생"과 "판단(BUY/SELL/HOLD)"이
> 별도 단계로 분리되어 있었으나, 이 둘은 하나의 의사결정 프로세스로 통합하는 것이 적절하다.
> 분리 시 책임 경계가 모호해지고 상태 관리 복잡도만 증가한다.

## 2. 이해관계자

| 역할 | 담당 | 책임 | 의사결정 권한 |
|------|------|------|---------------|
| 프로젝트 오너 (PO) | 사용자 본인 | 요구사항 정의, 매매 전략 방향 결정, 각 Phase 승인 | 범위 변경, 전략 선택, 실전 전환 결정 |
| 수석 아키텍트 (SA) | Claude | 시스템 설계, 기술 가이드, 코드 작성, 산출물 작성, 코드 리뷰 | 기술 선택, 아키텍처 결정, 구현 방식 |

### 커뮤니케이션 규칙

- 각 Phase 산출물 완성 시 PO 리뷰 요청
- PO의 "승인" 또는 "수정 요청" 후 다음 단계 진행
- 매매 전략 관련 결정은 반드시 PO가 최종 승인
- 기술적 결정(라이브러리 선택, 설계 패턴 등)은 SA가 제안하고 PO에게 설명 후 진행

## 3. 프로젝트 범위

### 3.1 IN SCOPE (MVP에 포함)

| # | 기능 영역 | 상세 |
|---|-----------|------|
| S1 | 시장 데이터 수집 | KOSPI/KOSDAQ 종목의 일봉·실시간 시세·호가·거래량 수집 |
| S2 | 종목 유니버스 관리 | 매매 대상 종목 풀 정의 및 필터링 (시가총액, 거래량 등) |
| S3 | 매수 시그널 생성 | 기술적 분석 지표 기반 진입 시그널 생성 |
| S4 | 매도 시그널 생성 | 손절/익절/트레일링스탑 기반 청산 시그널 생성 |
| S5 | 포트폴리오 관리 | 종목 추가/제거, 비중 배분, 포지션 상태 관리 |
| S6 | 리스크 관리 | 일일 최대 손실 제한, 최대 포지션 수 제한, 종목당 비중 제한 |
| S7 | 주문 실행 | 한국투자증권 REST API를 통한 매수/매도 주문 전송 및 체결 확인 |
| S8 | 백테스팅 | 과거 데이터 기반 전략 수익률 검증 |
| S9 | 페이퍼 트레이딩 | 한투 모의투자 환경에서 실전 시뮬레이션 |
| S10 | 매매 로깅 | 모든 시그널·주문·체결·포지션 변경 기록 |
| S11 | 성과 리포팅 | 수익률, 승률, MDD 등 핵심 지표 리포트 |
| S12 | 알림 | 매매 실행·에러 발생 시 Telegram 알림 |

### 3.2 OUT OF SCOPE (향후 확장 대상)

| # | 제외 항목 | 제외 근거 |
|---|-----------|-----------|
| X1 | 선물/옵션 매매 | 증거금 관리, 만기일 처리 등 복잡도 3배 이상 증가 |
| X2 | 해외 주식 매매 | 환율, 시차, 별도 API 필요 |
| X3 | AI/ML 예측 모델 | MVP 이후 데이터 축적 후 적용 |
| X4 | 뉴스/감성 분석 | 별도 NLP 파이프라인 필요 |
| X5 | HFT (고빈도 매매) | 인프라 요구사항이 완전히 다름 (코로케이션 등) |
| X6 | 모바일 앱 | CLI/데스크톱 우선, 모바일은 알림으로 대체 |
| X7 | 웹 대시보드 | MVP에서는 로그 파일 + 텔레그램으로 대체 |

### 3.3 범위 변경 관리

범위 변경이 필요한 경우 다음 절차를 따른다:
1. PO가 변경 요청 제기
2. SA가 영향도 분석 (일정, 복잡도, 리스크)
3. PO-SA 합의 후 프로젝트 차터 버전 갱신
4. 영향받는 하위 산출물 업데이트

## 4. 가정사항 및 전제조건

### 4.1 가정사항 (Assumptions)

| # | 가정 | 불충족 시 영향 |
|---|------|----------------|
| A1 | 한국투자증권 계좌가 개설되어 있거나 개설할 수 있다 | API 연동 불가 → 다른 증권사로 전환 필요 |
| A2 | 한투 OpenAPI 신청 및 APP Key 발급이 가능하다 | API 사용 불가 |
| A3 | 개발 환경(PC)에 Python 3.11+을 설치할 수 있다 | 개발 환경 대안 필요 |
| A4 | 개발 기간 중 한투 API의 인터페이스 스펙이 크게 변하지 않는다 | API 연동 모듈 재작업 |
| A5 | PO가 매매 전략 방향(어떤 기술적 지표를 사용할지)을 Phase 1에서 결정할 수 있다 | 요구사항 정의 지연 |
| A6 | 매매 대상은 보통주에 한정한다 (ETF, ETN, 우선주 등은 MVP에서 제외) | 유니버스 관리 로직 변경 필요 |

### 4.2 전제조건 (Prerequisites)

Phase 1 진입 전 반드시 확인되어야 할 항목:

- [ ] 한국투자증권 계좌 보유 여부 확인
- [ ] 한투 OpenAPI 서비스 신청 여부 확인
- [ ] APP Key / APP Secret 발급 여부 확인
- [ ] 모의투자 신청 여부 확인
- [ ] 개발 환경 (OS, Python 버전) 확인

## 5. 기술 제약조건

### 5.1 확정 사항

| 항목 | 결정 | 근거 |
|------|------|------|
| MVP 대상 시장 | 국내 주식 현물 (KOSPI/KOSDAQ) | 선물/옵션 대비 단순, 증거금 관리 불필요 |
| 증권사 | 한국투자증권 | REST API 제공, 문서화 양호, 모의투자 지원 |
| 개발 언어 | Python 3.11+ | PO 숙련도, 금융 데이터 라이브러리 생태계 풍부 |
| 개발자 수준 | Python 기초 | 코드에 주석 충분히, 모듈을 작게 유지, 복잡한 패턴 지양 |

### 5.2 기술 스택 (권장 - Phase 2 설계 시 확정)

| 계층 | MVP 기술 | 확장 시 전환 대상 | 선택 근거 |
|------|----------|-------------------|-----------|
| 데이터 저장 | SQLite | PostgreSQL | 설치 불필요, 파일 기반, Python 내장 지원 |
| 설정 관리 | YAML 파일 | - | 가독성 좋음, 수정 용이 |
| 스케줄링 | APScheduler | Celery | 경량, Python 네이티브, 별도 브로커 불필요 |
| HTTP 클라이언트 | requests | httpx (async) | 단순 명료, 디버깅 용이 |
| 기술적 분석 | TA-Lib 또는 pandas-ta | - | 표준 지표 라이브러리 |
| 백테스팅 | 자체 경량 구현 | Backtrader / Zipline | Backtrader 러닝커브 높음, MVP에 과도 |
| 알림 | Telegram Bot API | Slack, Discord | 무료, 모바일 즉시 알림, 구현 단순 |
| 로깅 | Python logging → 파일 | ELK Stack | 표준 라이브러리, 별도 인프라 불필요 |

### 5.3 한국투자증권 API 제약사항

| 제약 | 상세 | 시스템 대응 방안 |
|------|------|------------------|
| API 호출 제한 | 초당 호출 수 제한 존재 | Rate Limiter 구현, 호출 간격 관리 |
| 환경 분리 | 모의투자 / 실전투자 URL·토큰 분리 | 설정 파일로 환경 전환 (dev/prod) |
| 인증 방식 | OAuth 토큰 기반, 토큰 유효기간 제한 | 토큰 자동 갱신 로직 구현 |
| 장 운영시간 | 정규장 09:00~15:30, 시간외 별도 | 스케줄러가 장 시간에만 매매 루프 실행 |
| 데이터 지연 | 실시간 시세에 약간의 지연 존재 | 시장가 주문 우선, 슬리피지 감안 |

## 6. 리스크 관리

### 6.1 리스크 등록부

| ID | 리스크 | 영향도 | 발생확률 | 리스크 등급 | 대응 전략 | 구체적 대응 방안 |
|----|--------|--------|----------|-------------|-----------|------------------|
| R1 | 매매 전략 수익성 미검증 | 높음 | 높음 | **심각** | 완화 | 최소 1년 과거 데이터 백테스트 필수, 페이퍼 트레이딩 2주 이상 선행, 실전 전환 시 소액(100만원 이하)으로 시작 |

| R3 | 버그로 의도하지 않은 대량 매매 | 높음 | 중간 | **높음** | 예방 | 1회 주문 최대 금액 제한(설정값), 일일 총 매매금액 상한 설정, 주문 전 더블체크 로직 |
| R4 | 개발자 Python 숙련도 부족 | 중간 | 높음 | **높음** | 완화 | 모듈을 작게 분리(파일당 100줄 이내 목표), 코드에 한글 주석 충분히, 복잡한 디자인 패턴 지양 |
| R5 | 시장 급변 시 손절 지연 | 높음 | 낮음 | **중간** | 완화 | 손절 주문은 시장가로 실행, 장중 포지션 모니터링 주기 최소 1분 이내 |
| R6 | 한투 API 스펙 변경 | 중간 | 낮음 | **낮음** | 수용 | API 호출을 별도 어댑터 계층으로 격리, 변경 시 어댑터만 수정 |
| R7 | 시스템 비정상 종료 | 높음 | 중간 | **높음** | 완화 | 포지션 상태를 DB에 영속화, 재시작 시 마지막 상태에서 복구, 미청산 포지션 경고 |

## 10. 성공 기준

### 10.1 기능적 성공 기준

| # | 기준 | 측정 방법 |
|---|------|-----------|
| F1 | 시장 데이터를 정상 수집하고 DB에 저장한다 | 일봉 데이터 조회 API 호출 성공률 95% 이상 |
| F2 | 정의된 전략 규칙에 따라 시그널을 생성한다 | 백테스트에서 수동 계산 결과와 시그널 일치율 100% |
| F3 | 시그널에 따라 자동으로 매수/매도 주문이 실행된다 | 모의투자에서 시그널 발생 후 주문 전송 확인 |
| F4 | 손절/익절 규칙이 기계적으로 실행된다 | 페이퍼 트레이딩 중 손절/익절 조건 도달 시 100% 실행 |
| F5 | 모든 매매 내역이 로깅된다 | 로그 DB와 증권사 매매 내역 대조 일치 |

### 10.2 비기능적 성공 기준

| # | 기준 | 목표 수치 | 측정 방법 |
|---|------|-----------|-----------|
| NF1 | 시스템 가동률 (장중) | 99% 이상 | 장중 6.5시간 중 비가동 시간 4분 이내 |
| NF2 | 시그널→주문 지연 | 5초 이내 | 시그널 타임스탬프 ~ 주문 전송 타임스탬프 차이 |
| NF3 | 리스크 게이트 작동 | 100% | 일일 최대 손실 도달 시 추가 매매 차단 확인 |
| NF4 | 에러 알림 지연 | 1분 이내 | 에러 발생 ~ Telegram 수신 시간 차이 |

### 10.3 명확한 비(非)목표

본 시스템은 다음을 보장하지 않으며, 이를 목표로 설정하지 않는다:

- **"반드시 수익을 낸다"는 보장은 하지 않는다.**
  어떤 매매 전략도 100% 수익을 보장할 수 없다.
- **시스템이 할 수 있는 것**: 정의된 규칙을 감정 개입 없이 일관되게 실행하는 것.
- **시스템이 할 수 없는 것**: 미래 주가를 예측하는 것.
- **수익성은 전략의 품질에 의존한다.** 시스템은 전략을 실행하는 도구일 뿐이다.

## 12. 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| **ATS** | Automated Trading System. 본 프로젝트에서 개발하는 자동매매 시스템의 약칭 |
| **MVP** | Minimum Viable Product. 최소 기능 제품. 핵심 기능만 구현한 첫 번째 출시 버전 |
| **BRD** | Business Requirements Document. 비즈니스 관점의 요구사항 정의서 |
| **SRS** | Software Requirements Specification. 소프트웨어 관점의 요구사항 명세서 |
| **SAD** | Software Architecture Document. 소프트웨어 아키텍처 설계서 |
| **ERD** | Entity-Relationship Diagram. 데이터 모델의 엔티티 관계도 |
| **NFR** | Non-Functional Requirements. 비기능 요구사항 (성능, 보안, 가용성 등) |
| **PO** | Project Owner. 프로젝트 오너. 요구사항 결정 및 승인 권한자 |
| **SA** | Solution Architect / 수석 아키텍트. 기술 설계 및 구현 책임자 |
| **시그널** | 매매 판단의 트리거가 되는 이벤트. 진입 시그널(매수)과 청산 시그널(매도)로 구분 |
| **유니버스** | 매매 대상이 될 수 있는 종목의 전체 풀 (예: KOSPI200 구성 종목) |
| **포지션** | 현재 보유 중인 특정 종목의 매수 상태 (종목, 수량, 매수가, 상태 포함) |
| **손절 (Stop Loss)** | 손실이 사전 정의된 기준에 도달했을 때 포지션을 청산하는 것 |
| **익절 (Take Profit)** | 수익이 사전 정의된 목표에 도달했을 때 포지션을 청산하는 것 |
| **트레일링 스탑** | 주가 상승 시 손절 기준을 함께 올려가는 동적 손절 방식 |
| **백테스팅** | 과거 시장 데이터에 전략을 적용하여 가상 성과를 검증하는 과정 |
| **페이퍼 트레이딩** | 실제 돈을 사용하지 않는 모의 매매. 한투 모의투자 환경 활용 |
| **MDD** | Maximum Drawdown. 최대 낙폭. 고점 대비 최대 하락 비율 |
| **승률** | 전체 매매 중 수익으로 청산한 매매의 비율 |
| **슬리피지** | 의도한 주문 가격과 실제 체결 가격 간의 차이 |
| **리스크 게이트** | 주문 실행 전 리스크 조건을 확인하는 검증 단계 |
| **Phase Gate** | 다음 개발 단계로 넘어가기 위해 통과해야 하는 품질 검증 관문 |
| **한투** | 한국투자증권의 약칭 |
| **REST API** | HTTP 기반의 웹 API 방식. 한투가 제공하는 주문/조회 인터페이스 |
