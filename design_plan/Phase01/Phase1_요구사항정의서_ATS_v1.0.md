## 1. 비즈니스 목표

### 1.1 핵심 목표

본 시스템의 비즈니스 목표는 다음 세 가지이다:

| # | 목표 | 측정 지표 | 목표 수치 |
|---|------|-----------|-----------|
| BG-1 | 규칙 기반 일관된 매매 실행 | 시그널 발생 대비 실제 주문 실행률 | 95% 이상 |
| BG-2 | 손실 관리 자동화 | 손절 조건 도달 시 실행률 | 100% |
| BG-3 | 매매 성과 가시화 | 일별/주별/월별 성과 리포트 생성 | 자동 생성 |

### 1.2 비즈니스 컨텍스트

```
투자자 (PO)
    │
    │  ① 전략 설정 (어떤 지표, 어떤 파라미터)
    │  ② 리스크 한도 설정 (최대 손실, 최대 종목 수)
    ▼
┌──────────────────────────┐
│      ATS 시스템           │
│                          │
│  시장감시 → 시그널 → 매매 │
│                          │
│  투자자는 개입하지 않음    │
│  (알림만 수신)            │
└──────────┬───────────────┘
           │
           │  ③ 주문 전송
           ▼
    한국투자증권 API
           │
           │  ④ 체결 결과
           ▼
    매매 로그 / 알림
```

**핵심 원칙**: 시스템이 가동된 후 투자자는 **설정 변경**과 **모니터링**만 한다.
장중 수동 개입은 하지 않는 것을 원칙으로 한다. (긴급 상황 제외)

## 2. 매매 전략 정의

### 2.1 전략 개요

| 항목 | 결정 |
|------|------|
| 매매 스타일 | 스윙 트레이딩 (보유 기간 2~10 영업일) |
| 종목 유니버스 | KOSPI200 구성 종목 (대형주 중심) |
| 시그널 방식 | 기술적 분석 지표 조합 |
| 매매 방향 | 매수(Long)만 (공매도 없음) |

### 2.2 기술적 지표 및 적용 전략

> **아키텍트 노트**: PO가 4개 지표 모두에 관심을 표했으나,
> MVP에서는 **핵심 2개(이동평균선 + RSI)**로 시작하고,
> 백테스트 검증 후 나머지(볼린저밴드, MACD, 거래량)를 순차 추가하는 것을 권장한다.
>
> **근거**:
> - 4개 지표를 동시에 조합하면 파라미터 공간이 폭발적으로 넓어져 최적화가 어렵다
> - 각 지표의 개별 기여도를 분리 평가할 수 없다
> - 구현 복잡도를 낮춰 빠르게 "동작하는 시스템"을 만드는 것이 MVP의 핵심이다

#### MVP 전략 (v1.0): 이동평균선 + RSI 조합

```
[매수 조건] - 아래 조건이 모두 충족될 때
  ① 이동평균선: 단기(5일) MA가 장기(20일) MA를 상향 돌파 (골든크로스)
  ② RSI(14일)가 30~50 구간에 있음 (과매도에서 회복 중)
  ③ 당일 거래량이 20일 평균 거래량 이상 (거래량 확인용, 별도 지표 아님)

[매도 조건] - 아래 조건 중 하나라도 충족될 때
  ① 손절: 매수가 대비 -3% 하락
  ② 익절: 매수가 대비 +7% 상승
  ③ 트레일링 스탑: 보유 중 최고가 대비 -3% 하락
  ④ 이동평균선: 단기(5일) MA가 장기(20일) MA를 하향 돌파 (데드크로스)
  ⑤ RSI(14일)가 70 이상 (과매수 구간 진입)
  ⑥ 시간 제한: 보유 10영업일 초과 시 강제 청산
```

#### 확장 전략 (v2.0 이후 - OUT OF SCOPE)

| 우선순위 | 추가 지표 | 적용 방식 | 추가 시점 |
|----------|-----------|-----------|-----------|
| 1순위 | MACD | 매수 시그널 보조 확인 (MACD 히스토그램 양전환) | MVP 백테스트 완료 후 |
| 2순위 | 볼린저밴드 | 하단 밴드 터치 시 매수 후보, 상단 밴드 터치 시 익절 보조 | v2.0 |
| 3순위 | 거래량 급증 | 평소 대비 300% 이상 거래량 시 시그널 강도 가중 | v2.0 |

### 2.3 전략 파라미터 (설정 가능)

모든 파라미터는 설정 파일(YAML)로 관리하며, 코드 수정 없이 변경 가능해야 한다.

| 파라미터 | 기본값 | 설명 | 범위 |
|----------|--------|------|------|
| `ma_short_period` | 5 | 단기 이동평균 기간 (일) | 3~20 |
| `ma_long_period` | 20 | 장기 이동평균 기간 (일) | 10~120 |
| `rsi_period` | 14 | RSI 계산 기간 (일) | 7~21 |
| `rsi_buy_lower` | 30 | RSI 매수 하한 | 20~40 |
| `rsi_buy_upper` | 50 | RSI 매수 상한 | 40~60 |
| `rsi_sell_upper` | 70 | RSI 매도 상한 (과매수) | 65~80 |
| `stop_loss_pct` | -3.0 | 손절 기준 (%) | -1 ~ -10 |
| `take_profit_pct` | +7.0 | 익절 기준 (%) | +3 ~ +20 |
| `trailing_stop_pct` | -3.0 | 트레일링 스탑 (고점 대비 %) | -1 ~ -10 |
| `max_holding_days` | 10 | 최대 보유 일수 | 5~30 |
| `volume_confirm_ratio` | 1.0 | 거래량 확인 배수 (20일 평균 대비) | 0.5~3.0 |

### 2.4 종목 유니버스 관리

#### 유니버스 정의
- **대상**: KOSPI200 구성 종목
- **업데이트 주기**: 분기 1회 (KOSPI200 정기 변경 시)
- **관리 방식**: DB 테이블에 종목 리스트 저장, 수동 또는 반자동 갱신

#### 유니버스 필터 (스캔 제외 조건)
아래 조건에 해당하는 종목은 스캔 대상에서 제외한다:

| # | 필터 | 기준 | 근거 |
|---|------|------|------|
| UF-1 | 거래 정지 종목 | 당일 거래 정지 | 매매 불가 |
| UF-2 | 관리 종목 | 관리 종목 지정 | 높은 리스크 |
| UF-3 | 극저 거래량 | 20일 평균 거래량 10,000주 미만 | 슬리피지 리스크 |
| UF-4 | 신규 상장 | 상장 후 60영업일 미만 | 데이터 부족 |

## 4. 기능 요구사항 목록

### 4.1 기능 요구사항 (Functional Requirements)

각 요구사항의 ID 체계: `FR-[영역코드]-[번호]`

#### 영역: 데이터 수집 (DC: Data Collection)

| ID | 요구사항 | 우선순위 | 상세 |
|----|----------|----------|------|
| FR-DC-01 | 일봉 데이터 수집 | 필수 | KOSPI200 전 종목의 OHLCV(시가/고가/저가/종가/거래량) 일봉 데이터를 한투 API로 수집한다 |
| FR-DC-02 | 과거 데이터 수집 | 필수 | 백테스팅을 위해 최소 1년치 과거 일봉 데이터를 수집하여 DB에 저장한다 |
| FR-DC-03 | 장중 현재가 조회 | 필수 | 보유 종목의 현재가를 장중 주기적으로 조회한다 |
| FR-DC-04 | 유니버스 목록 관리 | 필수 | KOSPI200 구성 종목 리스트를 DB에 저장·관리한다 |
| FR-DC-05 | 유니버스 필터링 | 필수 | 거래정지, 관리종목, 극저거래량, 신규상장 종목을 필터링한다 |

#### 영역: 시그널 엔진 (SE: Signal Engine)

| ID | 요구사항 | 우선순위 | 상세 |
|----|----------|----------|------|
| FR-SE-01 | 이동평균선 시그널 | 필수 | 단기/장기 이동평균선의 골든크로스/데드크로스를 감지한다 |
| FR-SE-02 | RSI 시그널 | 필수 | RSI 값을 계산하고 매수/매도 구간 진입을 감지한다 |
| FR-SE-03 | 거래량 확인 | 필수 | 매수 시그널 발생 시 거래량이 기준 이상인지 확인한다 |

| FR-SE-05 | 시그널 로깅 | 필수 | 모든 시그널 발생을 타임스탬프와 함께 DB에 기록한다 |

#### 영역: 포트폴리오 관리 (PM: Portfolio Management)

| FR-PM-04 | 포트폴리오 조회 | 필수 | 현재 보유 종목, 평가 손익, 비중 등을 조회한다 |

| FR-RM-04 | 연속 손절 감지 | 필수 | 연속 손절 횟수를 추적하고, 한도 도달 시 일시 매매를 중단한다 |

#### 영역: 주문 실행 (OE: Order Execution)

| FR-OE-02 | 매도 주문 전송 | 필수 | 한투 REST API를 통해 매도 주문을 전송한다 |
| FR-OE-03 | 주문 체결 확인 | 필수 | 주문 전송 후 체결 여부를 조회하여 확인한다 |

### UC-11: 매수 시그널 스캔

| 항목 | 내용 |
|------|------|
| **액터** | ATS 스케줄러 |
| **설명** | 유니버스 종목을 스캔하여 매수 시그널을 생성한다 |
| **사전 조건** | 일봉 데이터가 DB에 존재. 현재 보유 종목 수 < 최대치. 리스크 게이트 미차단 상태 |
| **사후 조건** | 매수 시그널 발생 시 UC-20(매수 주문 실행)으로 전달 |
| **트리거** | 장중 매 10분 간격 (09:10, 09:20, ..., 15:10) |

**기본 흐름:**
1. 스케줄러가 스캔 프로세스를 트리거한다
2. 리스크 매니저에게 현재 매매 가능 상태인지 확인한다
3. (불가 시) 스캔을 건너뛴다
4. 유니버스 중 현재 미보유 종목에 대해 루프를 돈다
5. 각 종목의 일봉 데이터로 기술적 지표(MA, RSI)를 계산한다
6. 매수 조건(골든크로스 + RSI 30~50 + 거래량 확인)을 검사한다
7. 조건 충족 종목에 대해 매수 시그널을 생성한다
8. 시그널을 DB에 기록한다
9. 포트폴리오 매니저에게 포지션 생성을 요청한다
10. 포지션 크기(매수 수량)가 결정되면 UC-20(매수 주문 실행)을 호출한다

**예외 흐름:**
- E1: 매수 가능 종목 수가 최대치 → 스캔만 수행하고 주문은 보류
- E2: 여러 종목 동시 시그널 → 시그널 강도 순으로 정렬, 상위부터 매수 가능 수만큼 처리

### UC-12: 보유종목 모니터링

| 항목 | 내용 |
|------|------|
| **액터** | ATS 스케줄러 |
| **설명** | 보유 종목의 현재가를 확인하여 청산 시그널을 검사한다 |
| **사전 조건** | ACTIVE 상태인 포지션이 1개 이상 존재한다 |
| **사후 조건** | 청산 조건 충족 시 UC-21(매도 주문 실행)으로 전달 |
| **트리거** | 장중 매 1분 간격 |

**기본 흐름:**
1. 스케줄러가 모니터링 프로세스를 트리거한다
2. DB에서 ACTIVE 상태 포지션 목록을 조회한다
3. 각 포지션의 종목에 대해 한투 API로 현재가를 조회한다
4. 각 포지션에 대해 청산 조건을 검사한다:
   - 손절 조건: 현재가 ≤ 매수가 × (1 + stop_loss_pct/100)
   - 익절 조건: 현재가 ≥ 매수가 × (1 + take_profit_pct/100)
   - 트레일링 스탑: 현재가 ≤ 보유 중 최고가 × (1 + trailing_stop_pct/100)
   - 데드크로스: 단기 MA < 장기 MA (이전 시점에서 반전)
   - RSI 과매수: RSI ≥ rsi_sell_upper
   - 시간 제한: 보유 일수 > max_holding_days
5. 조건 충족 시 청산 시그널을 생성하고 DB에 기록한다
6. UC-21(매도 주문 실행)을 호출한다

**예외 흐름:**
- E1: 현재가 조회 실패 → 직전 조회 가격으로 판단, 에러 로그 기록, 다음 주기에 재시도
- E2: 여러 청산 조건 동시 충족 → 가장 먼저 감지된 조건을 청산 사유로 기록

|------|------|
| **액터** | 투자자 (PO) |
| **설명** | 과거 데이터에 전략을 적용하여 가상 성과를 검증한다 |
| **사전 조건** | 최소 1년치 일봉 데이터가 DB에 존재한다 |
| **사후 조건** | 백테스트 결과(수익률, 승률, MDD 등)가 출력된다 |

**기본 흐름:**
1. 투자자가 CLI에서 백테스트 명령을 실행한다 (기간, 전략 파라미터 지정)
2. 시스템이 지정 기간의 일봉 데이터를 DB에서 로드한다
3. 첫 영업일부터 마지막 영업일까지 순서대로 시뮬레이션한다:
   - 각 날짜마다 시그널 엔진을 호출하여 매수/매도 시그널을 확인한다
   - 시그널에 따라 가상 매수/매도를 실행한다 (체결가 = 다음 날 시가)
   - 포트폴리오 규칙, 리스크 규칙을 동일하게 적용한다
4. 시뮬레이션 완료 후 성과 지표를 계산한다:
   - 총 수익률, 연환산 수익률, 승률, 평균 수익/손실
   - MDD, 샤프비율, 매매 횟수
5. 개별 매매 내역과 성과 지표를 출력한다
