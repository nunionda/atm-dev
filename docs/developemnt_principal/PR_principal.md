1. PR을 생성하는 이유 (The Why)
PR은 단순히 코드를 병합하는 절차가 아니라, 팀의 기술 부채를 관리하고 지식을 공유하는 소통의 창구입니다.

코드 품질 보장 (Code Quality): 동료의 검토를 통해 잠재적인 버그를 발견하고, 더 나은 로직이나 컨벤션을 적용할 기회를 얻습니다.

지식 전파 (Knowledge Sharing): 내가 작성한 코드의 의도를 공유함으로써 팀원들이 프로젝트의 변화를 이해하고 학습하게 합니다.

히스토리 추적 (Traceability): 특정 기능이 왜, 어떻게 수정되었는지에 대한 기록을 남겨 향후 유지보수 시 강력한 근거가 됩니다.

CI/CD 자동화: PR 생성 시점에 자동화된 테스트와 빌드를 실행하여 메인 브랜치의 안정성을 깨뜨리지 않도록 보호합니다.

2. 지켜야 할 5가지 핵심 원칙 (The Principles)
① 하나의 PR에는 하나의 주제만 (Atomicity)
Bad: 로그인 기능 구현 + UI 버그 수정 + 라이브러리 업데이트를 한 PR에 담기.

Good: 기능을 쪼개서 리뷰어가 한 번에 이해할 수 있는 크기로 유지하세요. (권장: 200~300라인 이내)

② 명확한 제목과 설명 (Contextualization)
제목: [Feature] 로그인 API 연동 처럼 접두어를 활용해 성격을 명시하세요.

설명: "무엇을(What)" 했는지보다 "왜(Why)" 이렇게 구현했는지, 어떤 고민이 있었는지를 적으세요. 관련 이슈 번호(예: #123)를 연결하는 것은 필수입니다.

③ 셀프 리뷰는 필수 (Self-Review)
PR을 올린 직후, 본인이 직접 코드를 다시 확인하세요.

오타, 불필요한 console.log나 주석, 포맷팅 실수 등은 리뷰어의 시간을 뺏는 행위입니다.

④ 시각적 자료 활용 (Visual Aids)
UI/UX 변경이 포함된다면 반드시 **스크린샷이나 GIF(또는 영상)**를 첨부하세요. 리뷰어가 코드를 실행해보지 않고도 직관적으로 결과를 알 수 있게 해야 합니다.

⑤ 비판이 아닌 개선을 위한 피드백 (Empathy)
작성자: 리뷰어의 지적을 공격이 아닌 '코드 개선의 기회'로 수용하세요.

리뷰어: "이거 틀렸어요" 대신 "이 방식은 이런 문제가 생길 수 있는데, ~로 바꾸면 어떨까요?"라는 대안을 제시하세요.
